你好！我是你的老搭档，那个在数据治理领域摸爬滚打了50年的“老法师”。

做数据清洗（Data Cleaning）和异常处理（Exception Handling），最忌讳的是把“垃圾场”直接搬给客户看。客户看不懂 HTTP 500 报错，也看不懂 SQL 异常。

我们要设计的是一个**“数据检疫中心” (Data Quarantine Center)**。

所有从源系统（DMS、APP、官网）进来的数据，如果体检不合格，**绝对不能进入 C360 正式库**，而是被扣押在“检疫区”。这个模块的任务，就是让管理员决定：**是治好它（清洗/补全），还是扔掉它（丢弃），还是合并它（去重）。**

以下是基于 **Customer 360 Phase I** 的深度 PRD，涵盖了你没想到的极端边缘情况。

# 📘 产品需求文档 (PRD)：C360 数据异常与治理中心

## 1. 核心设计哲学 (Design Philosophy)
*   **不脏主库**：异常数据停留在 Staging（中间层）或 Quarantine（隔离表），只有修复后才能进入 Master Data（主数据）。
*   **授人以渔**：不仅要修现在的错，还要防止明天的错（规则反哺）。
*   **保留证据**：任何人工修改必须有“快照”，防止修坏了没法回滚。

## 2. 异常类型全景定义 (Exception Taxonomy)

在清洗过程中，必须捕获以下 **5 大类** 异常，并在界面上用不同颜色/图标区分：

| 异常大类 | 细分类型 | 场景举例 | 处理策略 (UI 动作) |
| :--- | :--- | :--- | :--- |
| **A. 有效性异常**<br>(Validity) | **格式错误** | 手机号含横杠、邮箱无@、身份证位数不对。 | **[清洗/编辑]** 支持正则自动修复建议 |
| | **值域越界** | 车龄 200 年、生日是未来时间、价格为负数。 | **[修正]** 或 **[置空]** |
| | **枚举非法** | 性别由源系统传来 "M/F"，目标系统要求 "0/1"。 | **[映射]** 建立值映射关系 |
| **B. 唯一性异常**<br>(Uniqueness) | **疑似重复** | 姓名+手机号高度相似，OneID 算法置信度为 80%（不敢自动合）。 | **[合并]** 进入比对工作台 |
| | **主键冲突** | 源系统推了两条 ID 一样但内容不同的数据。 | **[覆盖/丢弃]** 选择最新一条 |
| **C. 完整性异常**<br>(Completeness) | **关键缺失** | 有客户名但无任何联系方式（僵尸数据）。 | **[补录]** 或 **[归档]** |
| | **孤儿数据** | 有“订单记录”但找不到对应的“客户ID”。 | **[关联]** 手动挂载到某客户 |
| **D. 关联性异常**<br>(Consistency) | **逻辑互斥** | 身份证显示是女性，性别字段却是“男”。 | **[信任选择]** 选身份证还是选性别？ |
| | **状态冲突** | DMS 说“已交车”，CRM 说“未成交”。 | **[信任源]** 指定以 DMS 为准 |
| **E. 合规性异常**<br>(Compliance) | **授权缺失** | 数据进来了，但没有勾选“隐私协议”字段。 | **[冻结]** 暂时保留但不可营销 |

## 3. 功能模块详细设计

### 3.1 异常概览大屏 (Quarantine Dashboard)
*给老板和组长看的，监控数据健康度。*

*   **实时流监控**：显示今日从各源系统（DMS/BDC/APP）流入了多少数据，多少进入了隔离区。
    *   *隐喻*：像水处理厂的过滤池，多少是净水，多少是污水。
*   **阻断排行榜**：
    *   Top 1 错误类型：例如“手机号格式错误”占比 60%。
    *   Top 1 错误来源：例如“DMS 系统”贡献了 80% 的脏数据。
    *   *价值*：用来倒逼源系统整改。

### 3.2 异常治理工作台 (Governance Workbench) —— *核心操作页*

#### A. 筛选与任务分配
*   **视图模式**：支持按“错误类型”分组查看，也支持按“源系统”分组查看。
*   **智能分配**：系统自动将“DMS 的数据错误”分配给负责 DMS 对接的专员（避免一人处理所有）。

#### B. 列表字段设计
*   **严重级**：🔴 阻断 (无法入库) | 🟡 警告 (可入库但有瑕疵)
*   **数据摘要**：陈建华 | 138****0000 | BMW X5
*   **异常描述**：`[格式错误] 手机号包含非法字符 "-"`
*   **操作**：`修复` | `合并` | `忽略` | `丢弃`

### 3.3 详情页：三大修复模式 (The Repair Modes)

#### 模式一：格式/逻辑修复器 (The Editor)
*针对类型 A、C、D、E*

*   **界面布局**：
    *   **左侧（原罪）**：显示源系统发来的原始 JSON，标红错误字段。
    *   **右侧（救赎）**：表单编辑器。
*   **智能辅助功能**：
    *   **一键清洗建议**：如果手机号是 `138-1234-5678`，系统在输入框下方提示气泡：`💡 建议去除非数字字符 -> 13812345678`。点击即采纳。
    *   **字典映射**：如果报错是 `性别: Male`，系统提供下拉框，客户选择 `男`，并勾选 `[x] 将 Male 永久映射为 男`（下次自动修）。

#### 模式二：OneID 冲突仲裁庭 (The Merger)
*针对类型 B（疑似重复）*

*   **界面布局**：复用我们之前设计的 **“数据冲突解决工作台”**。
*   **特殊逻辑**：
    *   这是数据**入库前**的拦截。
    *   **操作 A（是同一个人）**：点击合并 -> 数据融合 -> 生成/更新同一个 OneID。
    *   **操作 B（不是同一个人）**：点击分离 -> 两条数据分别入库，生成两个不同的 OneID，并**加入“防重白名单”**（下次算法别再把这俩人算成一对了）。

#### 模式三：批量治理工具 (Bulk Actions)
*针对类型 C（大规模缺失）*

*   **场景**：DMS 导入了 1000 条历史数据，都没手机号。
*   **操作**：
    *   全选 1000 条。
    *   点击 **[批量处理]**。
    *   选择策略：`标记为“存量沉睡客户”并直接入库` 或 `全部丢弃`。

## 4. 关键交互逻辑与规则 (Business Rules)

这部分是 PRD 的灵魂，必须写给后端看。

### 4.1 "锁定与信任" 机制 (Locking & Trust)
**痛点**：今天管理员把陈建华的电话修好了，明天 DMS 又推了一遍错的电话过来，覆盖了正确的，怎么办？

*   **规则**：凡是在“异常中心”人工修改过的字段，必须打上 **`IS_MANUAL_CORRECTED = 1`** 标记。
*   **逻辑**：T+1 跑批时，ETL 脚本检查该标记。如果字段被人工锁定，**拒绝源系统的更新**，除非源系统的值与人工修改的值一致。

### 4.2 规则学习与反哺 (Rule Learning)
**痛点**：不能每天都让人修同一个错误。

*   **规则**：当管理员在“映射修复”中勾选了“永久映射”后。
*   **逻辑**：系统自动生成一条 **ETL 转换规则**（如 `Replace '-' with ''`）。下次数据进来，在进入异常中心之前，就先被这个规则自动洗干净了。

### 4.3 审计与回滚 (Audit Trail)
*   **记录**：谁（User）、改了什么（Old/New Value）、原因（Reason）。
*   **回滚**：提供“撤销治理”按钮，恢复到原始脏数据状态（用于误操作）。

## 5. 风险控制与边界 (Risk Management)

| 风险点 | 描述 | 解决方案 |
| :--- | :--- | :--- |
| **数据积压** | 异常数据太多，处理不过来，导致 C360 没有最新数据。 | **TTL 机制**：设置保留期限（如 30 天）。超过 30 天未处理的“低优异常”，自动执行默认策略（如：仅入库基础字段或直接丢弃）。 |
| **误杀** | 算法把两个不同的人判定为疑似重复，管理员手滑合错了。 | **分离工具**：提供“OneID 拆分”功能（Phase II），通过日志回溯还原。 |
| **性能瓶颈** | 异常中心积累了百万级脏数据，查询卡顿。 | **冷热分离**：只展示最近 7 天的异常任务。历史异常归档到离线表。 |

---

## 6. MVP 交付范围建议 (Phase I Scope)

为了赶进度，Phase I 不要试图做全自动 AI 清洗。

1.  **必须有**：
    *   异常列表页（能看到错）。
    *   基础编辑器（能改错）。
    *   合并比对页（能处理 OneID 冲突）。
    *   “忽略/丢弃”按钮（处理不了的先扔掉，保证流程通畅）。
2.  **可以砍**：
    *   批量修改功能。
    *   规则自动学习（反哺 ETL）。
    *   复杂的智能清洗建议（AI 提示）。

### 💡 给 UI 设计师的关键词 (UI Keywords)

*   **色系**：使用 **琥珀色/橙色 (Warning)** 作为主色调，区别于 C360 主系统的蓝色。提醒客户这里是“问题处理区”。
*   **Diff 视图**：大量使用 `Before` vs `After` 的对比样式（左红右绿）。
*   **空状态**：当没有异常时，显示一个大大的绿勾，文案“系统健康度 100%”，给客户正向反馈。

这份 PRD 结合了数据治理的底层逻辑（ETL/OneID）和前端交互体验，足以应对最挑剔的技术评审。
